Redefining the "presentational" tags, also know as "Cliff Richard tags".

The single most revolting "advance" in HTML5 is its attempt to redefine the archaic presentational tags. It is still in development (therefore aren't ideal and messy). It is recommended to avoid these tags whenever possible (esp. old browsers can't read them anyway, and 'em' does the job perfectly more often than not). However, it is a personal call, because some do use them.

'hr':

'u': no longer "underline", but unarticulated. It's also "useless" but bonus point for the abbreviation remaining intact.

'b': no longer "bold"; but "bollocks", or "attention".

'sub' and 'sup' are still subscript and superscript; but somehow not presentational anymore.

Conditional Comments - for older version of IE (up to version 9). IE10 onwards no longer need them.

It has become a common practice of latching extra CSS to a document to plaster over holes in those inept and naughty older IE versions, as example below:


                <link href="everything.css" rel="stylesheet">
                <!--[if IE]><link href="stupidie.css" rel="stylesheet"><![endif]-->

You can also target specific versions of IE:

  • <!--[if IE6>...
  • <!--[if IE7>...
  • <!--[if IE8>...
  • <!--[if IE9>...

Or, target versions that are greater than or less than a certain number:

  • <!--[if IE gt 6]>... which is greater than version 6
  • <!--[if IE gte 8]>... greater than or equal to
  • <!--[if IE lt 7]>... less than
  • <!--[if IE ite 7]>... less than or equal to

There are actually more options, such as M$'s own take on it, but largely totally unnecessary.

Like all hacks, it is best to avoid when possible. It's more for small fallbacks so that you can use the scrumptious likes of CSS3 without compomise. And don't assume you have to accommodate every stone-age versoin of IE - try to figure out what is sensible for you to support, by checking your website users' browser versions.

Accessible Forms: using Labels, Field sets and Legends, to allow disabled people to input into the forms.

Example of Label tag using 'for' attribute

Labels have the added bonus of visual browsers rendering the labels themselves clickable, putting the focus on the associated form field. Both 'name' & 'id' attributes are typically required: the 'name' for the form to handle that field and the 'id' for the label to associate it to.

You can group fields using the 'fieldset' tag, and using 'legend' tag inside the 'fieldset' tag for caption. BTW, I believe that it can be use even with abled-bodied users as well, visually.

HTML Forms Part 1: Input Types

Warning: none of these would work on IE version 9 and below. Even IE 10 failed to support some input types

HTML 5 greatly advances form controls, with numerous additional input types, several new attributes, and a handful of extra elements.

Below are the basic form fields created using the 'input' element, including:

  • text
  • password
  • checkbox
  • radio
  • submit
  • search: used for search query text box, performs exactly as a standard text input should. The main intention is one of style, using a CSS attribute selector: "input[type=search] {background: url(magnifyingglass.png) right no-repeat) }"
  • Other types includes: 'tel' for telephone; 'url' for web address; 'email' for email address; and CSS3 pseudo classes of ':valid' and ':invalid' to style the fileds
